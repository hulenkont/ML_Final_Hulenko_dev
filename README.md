# ☔ Двоканальний сервіс прогнозування опадів на основі Open-Meteo API & Machine Learning

## 📝 Опис та призначення проєкту
[cite_start]Цей проєкт є інтерактивним вебзастосунком на базі Streamlit, що реалізує прототип розумного міні-сервісу для автоматизованого збору метеоданих та довгострокового моделювання погоди[cite: 2, 58]. 

Головна відмінність цього застосунку — **двоканальна ML-архітектура (Multi-task Pipeline)**. Застосунок не просто передбачає факт дощу за допомогою класифікації, а й паралельно розраховує точний обсяг майбутньої вологи в міліметрах за допомогою регресії, порівнюючи результати власного ШІ з глобальною фізичною моделлю погоди.

---

## 🎯 Мета моделювання
Головне завдання сервісу — побудувати та оптимізувати дві незалежні цільові змінні:
1. [cite_start]**Канал класифікації (Факт опадів):** Бінарний target ($0$ — опадів немає, $1$ — опади є), що розраховується за умовою $precipitation\_sum > 0 \rightarrow 1$[cite: 36, 37, 40].
2. **Канал регресії (Обсяг опадів):** Кількісний target (чисте значення `precipitation_sum` в мм), що передбачає фізичну силу очікуваного шторму чи зливи.

## 📡 Використовувані дані та джерела (Data Engineering)
[cite_start]Усі погодні метрики завантажуються в реальному часі через HTTP-запити до безкоштовного європейського метеорологічного сервісу **Open-Meteo**[cite: 4, 15].

* [cite_start]**Історичний архів:** Завантаження щоденних (daily) даних за останні 2 роки через *Historical Weather API* для формування репрезентативного тренувального датасету[cite: 15, 17].
* [cite_start]**Оперативний прогноз:** Завантаження синоптичних карт на 1–14 днів уперед через *Forecast API*[cite: 53].

### Перелік вхідних ознак датасету (14 параметрів):
* [cite_start]`temperature_2m_max`, `temperature_2m_min`, `temperature_2m_mean` — температурні режими дня[cite: 21].
* [cite_start]`apparent_temperature_max`, `apparent_temperature_min` — показники відчутної температури[cite: 21].
* [cite_start]`wind_speed_10m_max`, `wind_gusts_10m_max` — максимальна швидкість та пориви вітру[cite: 21].
* [cite_start]`wind_direction_10m_dominant` — переважний кутовий напрямок вітру[cite: 21].
* [cite_start]`shortwave_radiation_sum` — сумарна сонячна короткохвильова радіація[cite: 21].
* [cite_start]`sunshine_duration`, `daylight_duration` — тривалість сонячного сяйва та світлового дня[cite: 21].
* [cite_start]`et0_fao_evapotranspiration` — рівень потенційного випаровування вологи грунтом[cite: 21].
* **Feature Engineering:** На основі часового індексу автоматично розраховуються змінні `month` (місяць) та `day_of_year` (день року) для врахування кліматичної сезонності.

[cite_start]⚠️ **Захист від витоку даних (Data Leakage):** Показник `precipitation_sum` примусово видаляється з матриці ознак $X$ перед початком навчання моделей класифікатора та регресора[cite: 43].

## 🏗️ Структура та архітектура ML-пайплайну

### 1. Валідація часових рядів (Time-Series Split)
Оскільки метеодані мають часову залежність, випадкове перемішування (Shuffle) призвело б до витоку інформації з майбутнього. [cite_start]Скрипт реалізує суворе хронологічне розбиття: останні **20% даних** відкладаються як повністю незалежний тестовий набір (`X_test`), а перші **80%** використовуються для крос-валідації через `TimeSeriesSplit(n_splits=5)`[cite: 32, 44].

### 2. Автоматизований відбір ознак (Feature Selection)
[cite_start]Перед передачею даних в алгоритми застосовується модуль `SelectFromModel`[cite: 22]. [cite_start]Він оцінює вагові коефіцієнти ознак «на льоту» та відсіває малозначущі параметри (шум), що упереджує перенавчання (overfitting)[cite: 22].

### 3. Оптимізація та стек моделей:
Програма виконує паралельний пошук найкращих параметрів за допомогою `GridSearchCV`:
* **Канал класифікації:** Змагаються ансамблевий метод `Random Forest Classifier` та лінійна `Logistic Regression` (з автоматичним масштабуванням через `StandardScaler`). [cite_start]Головна метрика — **F1-Score**[cite: 48, 49].
* **Канал регресії:** Оптимізуються `Random Forest Regressor` та лінійна регресія з регуляризацією `Ridge`. Оцінка якості здійснюється за метриками **MAE** (середня абсолютна помилка) та коефіцієнту детермінації **R²**.

## 🧭 Послідовність дій та інтерфейс користувача (Workflow)
[cite_start]Програма Streamlit розділена на три логічні взаємопов'язані етапи управління через Sidebar[cite: 59]:

### 1️⃣ Блок: Завантаження даних (Load data)
* Користувач гнучко задає локацію (через текстову назву міста або прямі координати) та часовий інтервал[cite: 24, 25, 60].
* [cite_start]Програма завантажує дані та виводить блоки **EDA (Розвідувального аналізу)**: кругову діаграму балансу класів та Box-plot розподілу обсягів дощів по місяцях[cite: 63].

### 2️⃣ Блок: Навчання ML моделі (ML model training)
* [cite_start]Одним натисканням запускається паралельний пошук по решітці для класифікаторів та регресорів[cite: 65].
* Інтерфейс виводить зведені інтерактивні таблиці метрик якості, теплову карту **Confusion Matrix (Матриці помилок)** та графік важливості відібраних ознак (Feature Importance)[cite: 51, 66].

### 3️⃣ Блок: Прогноз (Forecast)
* [cite_start]Найважливіший шар інтерфейсу, що формує індивідуальну картку погоди на сьогодні та зведену таблицю на 1–14 днів уперед[cite: 67, 68].
* [cite_start]**UI Бенчмаркінг:** Поруч виводяться два показники кількості опадів — розрахований нашою моделлю регресії та базовий прогноз від сервера Open-Meteo, а також фінальний графік динаміки температур[cite: 71, 72].
